# Portfolio Backtester Refactor Plan 2026-05-28

## Goal

Replace the current fragmented return-evaluation paths with one authoritative portfolio-level simulation engine.

## Why

Current backtests mix several incompatible evaluation modes:

- average selected forward return
- active-day average return
- target-hit rate via intrahold high
- single-position daily signal replay
- calendar-as-cash annualization

These are useful research views, but they should not be treated as the final truth for strategy deployment.

## New Base Module

Implemented foundation:

- `src/a_share_predictor/portfolio_backtester.py`

Current scope:

- portfolio cash accounting
- max concurrent positions
- next-trade-day open entry after signal date
- fixed holding-day exit at close
- transaction cost
- slippage
- daily NAV output
- realized trade ledger

This is intentionally minimal. It is a base layer, not yet the full production engine.

## Integration Order

### Step 1

Use `portfolio_backtester.py` as the only engine for:

- `scripts/backtest_strategy_model_top10.py`

Reason:

- This script already has the closest shape to a portfolio process.
- It produces daily candidate sets and a daily equity curve, so migration cost is lowest.

### Step 2

Refactor these modules to output standardized candidate signals instead of self-contained performance summaries:

- `src/a_share_predictor/market_backtest_runner.py`
- `scripts/backtest_market_regime_v3.py`
- `scripts/backtest_bull_market_rank_score.py`

Standard candidate schema should include:

- `market_date`
- `symbol`
- `candidate_priority`
- `candidate_strategy`
- optional diagnostics such as `model_score`, `rank_score`, `market_state`

## Required Next Enhancements

The new engine still needs these before it can become the authoritative production evaluator:

1. Position sizing policy
   - equal weight
   - volatility scaling
   - liquidity cap

2. Exit policy abstraction
   - fixed holding period
   - stop loss
   - trailing stop
   - end-of-day or multi-day hybrid exits

3. A-share tradability constraints
   - limit-up / limit-down non-executable states
   - one-word board handling
   - minimum liquidity filter
   - gap-risk controls

4. Risk controls
   - sector cap
   - per-day turnover cap
   - gross exposure scaling by regime
   - max drawdown pause

5. Reporting
   - turnover
   - holding concentration
   - cash utilization
   - annualized return
   - max drawdown
   - Sharpe-like daily return stats

## Migration Rule

After migration, old modules should still be allowed to compute research diagnostics, but not final annualized return claims.

Only the portfolio engine should be allowed to answer:

- annualized return
- max drawdown
- ending equity
- capital utilization
- deployable trade count

## Success Criteria

The refactor is complete only when:

- all main strategy backtests produce candidate signals first
- all performance claims are generated by the unified portfolio engine
- strategy comparisons are based on identical capital, cost, and execution assumptions
